    In build_dispatch_table() — inside the loop over unique_ops, add a branch on binding style:

    To detect RPC style, look up the binding for each operation. Thread the binding reference through
    the unique_ops collection: change the Vec element type from (String, String) to
    (String, String, BindingStyle, Option<String>) — (op_name, soap_action, style, rpc_namespace).

    In the binding_op collection loop, read:
      let style = binding.soap_binding.style.clone();
      let rpc_ns = if style == BindingStyle::Rpc {
          binding_op.input.body.namespace.clone()
              .or_else(|| Some(resolved.definition.target_namespace.clone()))
      } else {
          None
      };

    In the input_type resolution (replacing the existing resolve_input_element call):
    ```rust
    let input_type = if style == BindingStyle::Rpc {
        let ns = rpc_ns.as_deref().unwrap_or(&resolved.definition.target_namespace);
        Some(QName::new(ns, &op_name))
    } else {
        resolve_input_element(resolved, &op_name)
    };
    ```

    Add pub fn build_dispatch_table_for_service() as a new public function that accepts
    service_name: &str in addition to the existing parameters, and only collects binding ops
    from bindings that are referenced by that service's ports:

    ```rust
    pub fn build_dispatch_table_for_service(
        service_name: &str,

        handlers: HashMap<String, Arc<dyn SoapHandler>>,
        auth_bypass: &HashSet<String>,
        default_handler: Option<Arc<dyn SoapHandler>>,
    ) -> Result<DispatchTable, DispatchError>
    ```

    This function reuses the same logic as build_dispatch_table() but scopes the binding_op
    collection to the named service only. Factor the shared logic: extract a private helper
    collect_ops_for_service(service_name, resolved) -> Vec<(String, String, BindingStyle, Option<String>)>
    and use it from both build_dispatch_table (all services) and build_dispatch_table_for_service (one service).

    Preserve full backward compatibility: build_dispatch_table() signature is unchanged.

    In src/server.rs — extend SoapService to support multi-service routing.

    Add to SoapService:
    ```rust
    /// Per-service dispatch tables keyed by route path.
    /// Empty when single-service (backward-compat mode).
    service_tables: HashMap<String, Arc<DispatchTable>>,
    ```

    In ServerBuilder::build():
    1. After resolving the WSDL, check if resolved.definition.services.len() > 1.
    2. If multiple services: iterate services, for each service extract the port address URL path
       (use the path component of the address string — split on "/" after the host, or if no "//",
       treat the whole string as a path). Call build_dispatch_table_for_service() per service.
       Collect into service_tables HashMap<String, Arc<DispatchTable>>.
       The existing single dispatch_table field is populated from the first service (or remains
       from build_dispatch_table for backward compat).
    3. If single service: service_tables is empty, existing build_dispatch_table() is used as before.

    In SoapService::into_router():
    1. If service_tables is non-empty: register one route per entry in service_tables, each route
       being a POST handler that uses that service's DispatchTable.
       Pass the Arc<SoapService> as state (same as before) but the route closure must look up
       the correct DispatchTable from service_tables by the matched path.
    2. If service_tables is empty: existing single-route behaviour unchanged.

    For multi-service axum routing, use axum's path matching. Each service gets its own route:
    ```rust
    for (path, table) in &state.service_tables {
        router = router.route(path, post(soap_post_handler_for_table(...)));
    }
    ```

    The cleanest approach: have soap_post_handler accept a path extension or pass DispatchTable
    directly as additional state using axum's Extension layer or a wrapper state type.
    Recommended: create a thin SoapServiceRoute struct that holds Arc<SoapService> + Arc<DispatchTable>
    for the specific service, and use that as axum State for multi-service routes.

    IMPORTANT: The single-service path (service_tables empty) must work identically to today.
    Do not break any existing tests. The ServerBuilder API gains no new methods — multi-service
    detection is automatic from the WSDL.

    For the address path extraction helper: extract_path_from_url(address: &str) -> String:
    - If address contains "://", take everything after the third "/", prepend "/".
    - Otherwise return address as-is if it starts with "/", else prepend "/".

    For the integration tests: create a MULTI_SERVICE_WSDL constant with two services (ServiceA,
